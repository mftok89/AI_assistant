## CLAUDE.md の骨子（実装時に作成）

1. **役割**: 理さんの個人秘書。時間の最大化・判断負荷の最小化が使命
2. **行動原則**: 結論ファースト／確認より実行（宣言してから動く）／金銭・契約・外部送信のみ事前確認／推測で書かない

4. **PARA保存ルール**: 成果物の保存先判断基準

6. **MCP利用ルール**: メール送信はしない（下書きまで）、カレンダー登録は内容を提示してから実行、Notion タスクは追加・更新可（削除は事前確認）、など安全側の規定
7. グローバルCLAUDE.md（日本語応答・Obsidian記録）はそのまま生きるので重複記載しない

## Notion タスク DB（フェーズ2で1つ作成）

最初は **タスク DB 1つ** のみ。慣れたら会議メモ DB 等を追加検討する。

### DB 名

`AI秘書タスク`（Notion ワークスペース内に新規作成）

### プロパティ

| プロパティ名 | 型 | 選択肢 / 備考 |
|-------------|-----|--------------|
| 名前 | タイトル | タスク内容（必須） |
| ステータス | セレクト | `未着手` / `進行中` / `待ち` / `完了` |
| 優先度 | セレクト | `P1-今日` / `P2-今週` / `P3-後で` |
| 期限 | 日付 | なければ空欄 |
| ソース | セレクト | `メール` / `会議` / `プロジェクト` / `自分` / `その他` |
| メモ | テキスト | 補足・関連メール件名など |

### ビュー（Notion 側で手動作成）

- **今日**: ステータス ≠ 完了 かつ（期限 = 今日 OR 優先度 = P1-今日）
- **未完了一覧**: ステータス ≠ 完了、期限昇順
- **完了**: ステータス = 完了

### config/tools.md への記載（オンボーディング時）

- 会議メモ DB: （未作成・将来追加）

## プロトコル仕様（Notion 連携）

フェーズ1で `protocols/` を作成する際、以下の内容を `daily-briefing.md` と `email-triage.md` に含める。

### protocols/daily-briefing.md

**目的**: カレンダー・メール・Notion タスクを統合した朝ブリーフィング

**手順**:

1. **Google カレンダー**（MCP）: 今日の予定を取得
2. **Gmail**（MCP）: 未読・要対応メールを件数と概要で取得（本文全文は不要）
3. **Notion タスク DB**（MCP）: `config/tools.md` の DB を参照し、未完了タスクを取得
   - 優先度 `P1-今日` または期限が今日のものを「今日やること」に
   - 期限超過を「期限切れ」として先頭に警告
   - ソース `プロジェクト` のタスクは project-mgmt 登録分（プロジェクト納期）。README 走査は不要
4. **出力形式**（結論ファースト）:
   - 今日の予定（時刻順）
   - 今日やること（Notion タスク、P1 + 期限今日）
   - 期限切れタスク（あれば）
   - メール要約（未読件数 + 要対応 TOP3）
5. **保存**: 要約を `areas/daily-briefings/YYYY-MM-DD.md` に保存

**Notion フォールバック**: MCP 障害時は `areas/tasks-backup.md` があれば参照。なければ「Notion 未接続。タスク確認は手動で」と明示。

### protocols/email-triage.md

**目的**: メール4段階優先度分類 + 返信下書き + 要対応タスクの Notion 登録

**優先度分類**:

| 段階 | 基準 | 対応 |
|------|------|------|
| P1 | 今日中に返信・対応が必要 | 返信下書き作成 + Notion タスク（P1-今日） |
| P2 | 今週中 | 返信下書き or 返信方針提示 + Notion タスク（P2-今週） |
| P3 | 情報共有・後でよい | 要約のみ。タスク化は理さん確認後 |
| P4 | 不要・スパム | 一覧に載せるのみ |

**手順**:

1. **Gmail**（MCP）: 未読・直近24時間のメールを取得
2. 上記4段階で分類し、P1/P2 を先に提示
3. P1/P2 について **Gmail 下書き**を作成（送信はしない）
4. **Notion タスク DB**（MCP）: P1/P2 の各メールについて、重複がなければタスクを追加
   - 名前: `[メール] 件名の要約`
   - ステータス: `未着手`
   - 優先度: P1 → `P1-今日`、P2 → `P2-今週`
   - 期限: P1 → 今日、P2 → 今週金曜（なければ空欄）
   - ソース: `メール`
   - メモ: 送信者・件名・メール URL（取得可能なら）
5. 既存 Notion タスクと件名・送信者が重複する場合は **新規作成せず** 既存を案内

**Notion フォールバック**: MCP 障害時は `areas/tasks-backup.md` に追記し、「Notion 接続後に移行してください」と報告。

### protocols/project-mgmt.md（プロジェクト納期 → Notion 連携）

プロジェクト納期は朝ブリーフィングのデータ源（Notion タスク DB）経由で拾う。**`projects/*/README.md` の走査は daily-briefing には含めない**（タスクの正が Notion という設計に統一）。

**ルール**:

1. プロジェクト立ち上げ時・納期設定・納期変更時に、**Notion タスク DB へ納期タスクを登録または更新**する
2. タスク例:
   - 名前: `[PRJ-001] 案件名 — 納期`
   - ステータス: `未着手`（進行中なら `進行中`）
   - 優先度: 納期が今週以内 → `P2-今週`、それ以外 → `P3-後で`（納期当日は `P1-今日`）
   - 期限: プロジェクトの納期
   - ソース: `プロジェクト`
   - メモ: `projects/PRJ-001_案件名/README.md` への参照
3. プロジェクト完了・アーカイブ時: 対応する Notion タスクのステータスを `完了` に更新

これにより、daily-briefing の手順3（Notion 未完了タスク取得）だけで「PRJ-001 納期 4/15」も期限切れ警告に含まれる。
